# ARC — LLM Integration Spec

## How `project.arc` Becomes Your Codebase's Brain for AI

> This document specifies how `arc` generates a single file you upload to any LLM
> (Claude, Cursor, Copilot, GPT-4) that gives it complete understanding of your codebase —
> architecture, rules, features, data flows, conventions — so it can work correctly
> without explanation, and so `arc check` can validate everything it produces.
>
> **The goal:** upload `project.arc` to an LLM the same way you upload `schema.prisma`.
> The LLM immediately knows your entire codebase. No explanation needed.
> Everything it writes is validated by `arc check`.

## 1. The Problem Being Solved

### The current situation

Every time you start a new AI conversation about your codebase, you spend 5-10 minutes explaining:

- What your stack is
- Where things live
- What rules to follow
- What already exists
- What patterns to use
- What not to touch

The AI then writes code that:

- Puts logic in the wrong layer
- Imports Prisma directly in a component
- Creates a hook without `'use client'`
- Invents a new pattern instead of following your conventions
- Reimplements a feature that already exists

Then you spend time correcting it. The context decays mid-conversation and the AI starts drifting.

### The `schema.prisma` parallel

When you upload `schema.prisma` to an LLM, it immediately knows:

- Every table and its columns
- Every relationship
- Every constraint
- The exact types to use

It doesn't invent tables. It uses what exists. It follows the schema exactly.

**`project.arc` should do the same thing for your entire codebase.**

### What `arc` solves

```md
Without arc                          With arc
────────────────────────────         ────────────────────────────
"My app uses Next.js and             Upload project.arc
Supabase and..."                     ↓
"The actions folder has..."          LLM knows everything instantly
"Don't import Prisma in..."          ↓
"We use is_paid to gate..."          "Add a share feature"
"The hook should be called..."       ↓
5-10 min explanation                 LLM writes correct code
↓                                    ↓
LLM writes something wrong           arc check validates it
↓                                    ↓
You correct it                       Build passes or fails clearly
↓
Repeat
```

## 2. The Two Files That Power This

### `project.arc` — what you write and maintain

The source of truth. You write this once and keep it updated as your app grows.
Contains layers, rules, flows, conventions, features, context.
Human-readable. Version controlled. The `schema.prisma` of your architecture.

### `.arc/LLM_CONTEXT.md` — what you give to the AI

Auto-generated by `arc context`. Never edit manually.
A single markdown file optimized for LLM consumption.
Contains everything in `project.arc` rendered as structured prose + tables + code blocks.
Designed so any LLM can read it once and work correctly on your codebase.

**Regenerate whenever `project.arc` changes:**

```bash
arc context
# → writes .arc/LLM_CONTEXT.md
```

**Upload to LLM:**

- Claude: paste into conversation or use Projects
- Cursor: add to `.cursorrules` or paste as context
- Copilot: paste into chat
- Any LLM: paste the file contents as system context

## 3. New DSL Block: `features`

Add one new top-level block to the `.arc` DSL: `features`.

This is the "what exists in my codebase" block — analogous to how `schema.prisma` shows every table that exists. It tells the LLM what features are already built so it doesn't rebuild them and knows how to extend them.

### Syntax

```md
features {

  feature "<name>" {
    description  "<what it does>"
    status       built | in-progress | planned
    layer        <layer-name>
    files        ["<path>", ...]
    actions      ["<action-name>", ...]    # server action function names
    hooks        ["<hook-name>", ...]      # hook names
    components   ["<component-name>", ...]
    routes       ["<route-path>", ...]
    tables       ["<table-name>", ...]
    depends-on   ["<feature-name>", ...]
    notes        "<any extra info for the LLM>"
  }

}
```

### Real example

```md
features {

  feature "auth" {
    description  "Google OAuth via Supabase. Auto-creates profiles row on first sign-in."
    status       built
    files        ["src/lib/supabase.ts", "src/app/api/auth/callback/route.ts"]
    routes       ["/api/auth/callback", "/login", "/logout"]
    tables       ["auth.users", "public.profiles"]
    notes        "DB trigger handles profile creation — do not create profiles manually"
  }

  feature "notes-crud" {
    description  "Full CRUD on notes. Tags, pinning, public/private toggle, linked notes."
    status       built
    actions      ["getNotes", "createNote", "updateNote", "deleteNote", "togglePin"]
    hooks        ["useNotes"]
    components   ["NoteList", "NoteEditor", "NoteCard"]
    routes       ["/dashboard", "/note/[id]"]
    tables       ["public.notes"]
    depends-on   ["auth"]
  }

  feature "note-history" {
    description  "Version snapshots of note content. Manual and auto-triggered saves."
    status       built
    actions      ["createSnapshot", "getSnapshots", "restoreSnapshot"]
    components   ["HistoryPanel", "SnapshotCard"]
    tables       ["public.note_versions", "public.notes_history_snapshots"]
    depends-on   ["notes-crud"]
  }

  feature "payments" {
    description  "Polar.sh one-time purchase. Webhook sets profiles.is_paid = true."
    status       built
    files        ["src/lib/polar.ts", "src/app/api/webhooks/polar/route.ts"]
    actions      ["updatePaymentStatus", "getPaymentStatus"]
    routes       ["/upgrade", "/api/webhooks/polar"]
    tables       ["public.profiles"]
    notes        "is_paid is the gate — check this boolean, not current_plan"
  }

  feature "access-control" {
    description  "Protected routes check profiles.is_paid. Redirects to /upgrade if false."
    status       built
    files        ["src/app/(protected)/layout.tsx", "src/actions/profiles.ts"]
    depends-on   ["auth", "payments"]
    notes        "Gating logic lives in layout.tsx — do not gate in individual pages"
  }

  feature "public-notes" {
    description  "Notes with is_public=true are viewable without auth at /note/[id]"
    status       built
    components   ["PublicNoteView"]
    routes       ["/note/[id]"]
    depends-on   ["notes-crud"]
  }

  feature "feedback" {
    description  "In-app feedback, bug reports, and feature requests submitted to DB."
    status       built
    components   ["FeedbackModal", "BugReportForm"]
    tables       ["public.feedback", "public.bug_reports", "public.feature_requests"]
  }

  feature "sharing" {
    description  "Track share actions and share counts per URL."
    status       built
    tables       ["public.share_actions", "public.share_counts"]
  }

  feature "usage-tracking" {
    description  "Track conversion, download, export, and batch actions per user."
    status       built
    tables       ["public.usage_conversion_actions", "public.usage_download_actions", "public.usage_export_actions", "public.usage_batch_job_actions"]
  }

  feature "offline-mode" {
    description  "Allow notes to work offline and sync on reconnect."
    status       planned
    notes        "Not started. Will need IndexedDB + sync queue."
  }

}
```

### Status values

| Value | Meaning |
| --- | --- |
| `built` | Feature is complete and in production |
| `in-progress` | Being actively developed |
| `planned` | Decided but not started |

The LLM uses `status` to know:

- `built` → don't rebuild it, extend it, use the listed files/actions/hooks
- `in-progress` → be aware it exists partially, ask before touching
- `planned` → this is what the user will ask you to build next

## 4. New CLI Command: `arc context`

Add `arc context` to the CLI. This is the primary LLM integration command.

### What it does

Reads `project.arc` (full AST) and generates `.arc/LLM_CONTEXT.md` — a single markdown file structured for LLM consumption.

### Output format

```md
◆ arc context

  ✓  Generated .arc/LLM_CONTEXT.md

  Upload this file to your LLM before any coding session.
  The LLM will understand your full codebase without explanation.

  Size: 4.2kb — fits in any LLM context window.
```

### When to run

- After any change to `project.arc`
- After adding a new feature to `features {}`
- Before starting a new AI coding session
- Can be added to a git pre-commit hook

```markdown
2. Run `arc scaffold <type> <name>` to create the file

5. Client hooks go in `src/hooks/` — always `'use client'`, always named `use*`

| I'm writing... | Put it in |
|---|---|
| A database query | `src/actions/` (server action) |
| Client-side state | `src/hooks/` (custom hook) |
| A UI component | `src/components/app/` |
| A shared TypeScript type | `src/types/index.ts` |
| An API endpoint | `src/app/api/{path}/route.ts` |
| Environment config | `src/config/index.ts` |
| Infrastructure client | `src/lib/` |
```

## 6. LLM Instruction Preamble

The generated `.arc/LLM_CONTEXT.md` begins with a special instruction block that tells the LLM exactly how to behave. This is the most important part of the file.

This preamble is **hardcoded** — always prepended by `arc context`, not user-editable.

```markdown
# CODEBASE CONTEXT
> Generated by arc · project.arc · {timestamp}

## Instructions for the AI Reading This File

You are working on a codebase with strict architectural rules enforced by `arc check`.
Read this entire file before writing any code. Then follow these rules exactly:

1. **Architecture is enforced.** Every rule in this file is checked by `arc check`
   before the build runs. Code that violates these rules will fail the build.
   Do not write code that violates them — it will be rejected.

2. **Use what exists.** The "Features" section lists everything already built.
   Before creating anything, check if it exists. Use the listed actions, hooks,
   and components. Do not reimplement them.

3. **Follow the layer rules.** Every file belongs to a layer. Each layer has strict
   rules about what it can import. Check the layer table before writing any import.

4. **Follow naming conventions exactly.** The conventions table shows required filename
   patterns. Deviation from them causes `arc check` to fail.

5. **Directives are required.** Files in `src/actions/` must have `'use server'` as
   their first line. Files in `src/hooks/` must have `'use client'`. Missing these
   causes a build error.

6. **When in doubt, ask.** If a feature request is ambiguous about which layer
   it belongs in, state your reasoning before writing code.

7. **Types live in `src/types/`** — never define significant types inline in components
   or actions. Add them to the types layer.

8. **The database schema is in `prisma/schema.prisma`.** Do not invent tables or columns.
   Use what exists in the schema.

---
```

## 7. How `arc check` Closes the Loop

The LLM context file tells the LLM the rules. `arc check` enforces them after the LLM writes code.
This creates a closed feedback loop — the LLM can't silently break architecture.

### The full loop

```md
You: "Add a notes export feature"
         │
         ▼
LLM reads LLM_CONTEXT.md
Knows: export actions go in src/actions/
       components go in src/components/app/
       must use 'use server' in actions
       must not import Prisma in components
         │
         ▼
LLM writes code
         │
         ▼
arc check runs (automatically on npm run build)
         │
    ┌────┴────┐
    │         │
  Pass       Fail
    │         │
    ▼         ▼
Build      arc prints exact violations
succeeds   with file + line numbers
           │
           ▼
           You paste violations back to LLM
           "Fix these arc violations: ..."
           │
           ▼
           LLM fixes specifically
           │
           ▼
           arc check → Pass
```

### Feeding violations back to the LLM

When `arc check` fails after LLM-generated code, paste the output directly:

```md
User: "Fix these arc violations:"

  src/components/app/ExportButton.tsx:2
  └─ [import-boundary] [components] cannot import from [actions] → "@/actions/notes"

  src/actions/exportNotes.ts
  └─ [missing-directive] Missing "use server" — required in [actions] layer
```

The LLM immediately understands: the rule IDs are the same as in `LLM_CONTEXT.md`.
It knows exactly what to fix and why.

### Why this works better than instructions alone

LLMs following instructions occasionally drift, especially in long conversations.
`arc check` doesn't drift. It reads the actual files and applies the same rules every time.
The combination is more reliable than either alone:

- **Instructions alone:** LLM might forget a rule mid-conversation
- **arc check alone:** catches violations but doesn't prevent them
- **Both together:** LLM writes better code AND violations are caught before build

## 8. Versioning and Backward Compatibility

### Version field in `.arc`

The `arc` block has a `version` field. This is the version of the `project.arc` file itself,
not the project version. It enables backward compatibility as the DSL evolves.

```md
arc "note-app" {
  version "1.2.0"    ← arc file schema version
  ...
}
```

### Backward compatibility rules

These rules are permanent and must never be broken:

1. **Unknown keywords inside blocks are silently skipped** — old `.arc` files always parse
2. **New blocks are always optional** — `features {}` not present = features section omitted
3. **New fields inside blocks are always optional** — missing = use default value
4. **Removal of fields = major version bump** — never remove a field in a minor version
5. **The `arc` block version is informational** — the CLI never refuses to parse an old version

### Evolution strategy

When the DSL needs a new feature:

```md
# Adding a new field to layer block:
layer actions {
  path       "src/actions"
  new-field  "value"     ← new in arc 1.1.0
  ...
}
# Old arc versions: skip "new-field" silently — works fine
# New arc versions: parse and enforce "new-field" — works fine
```

```md
# Adding a new top-level block:
telemetry {              ← new in arc 1.3.0
  provider "datadog"
}
# Old arc versions: unknown top-level block skipped — works fine
# New arc versions: parsed and used — works fine
```

### Changelog block (optional, future)

For future use — documents breaking changes in the `.arc` file itself:

```md
# In project.arc (future feature, not implemented in v1):
# changelog {
#   "1.2.0" "Added features block and arc context command"
#   "1.1.0" "Added require filename and export shape rules"
#   "1.0.0" "Initial release"
# }
```

## 9. The Complete `project.arc` With Features

This is the full `project.arc` for the note-taking app — everything from the build spec
plus the new `features {}` block. This is what you maintain and upload to LLMs.

```md
# project.arc
# Single source of truth for architecture + features
# ─────────────────────────────────────────────────
# arc check   → enforce architecture rules
# arc map     → generate .arc/CODEBASE.md
# arc context → generate .arc/LLM_CONTEXT.md (upload to AI)
# arc print   → see full architecture summary

arc "note-app" {
  version  "1.0.0"
  stack    nextjs + supabase + prisma + polar + shadcn + tailwind
  language typescript
}

# ── Layers ────────────────────────────────────────────────────

layer types {
  path        "src/types"
  description "All TypeScript types derived from Prisma. No logic, no side effects."
  can import  []
  allow exports [type, interface, enum]
  forbid exports [function, class, const]
}

layer config {
  path        "src/config"
  description "Single access point for all process.env vars. Typed and validated."
  can import  [types]
}

layer lib {
  path        "src/lib"
  description "Prisma client singleton, Supabase client, Polar webhook helpers."
  can import  [types, config]
}

layer actions {
  path        "src/actions"
  description "Next.js Server Actions. All DB access lives here."
  can import  [types, config, lib]
  require directive "use server"
  require exports async-function
}

layer hooks {
  path        "src/hooks"
  description "React hooks. Client-side only. No direct DB."
  can import  [types, config]
  require directive "use client"
  require filename "use*.ts"
}

layer components {
  path        "src/components/app"
  description "App UI components. No Prisma. No Supabase. No direct DB."
  can import  [types, hooks, config]
}

layer ui {
  path        "src/components/ui"
  description "shadcn/ui. Never manually edit."
  can import  [types]
  readonly
}

layer app {
  path        "src/app"
  description "Next.js App Router: pages, layouts, API routes."
  can import  [types, config, hooks, components, actions, lib]
}

# ── Rules ─────────────────────────────────────────────────────

rule "no-prisma-outside-lib-actions" {
  severity    error
  description "Prisma client can only be imported in src/lib and src/actions"
  forbid import "@prisma/client"
  except in   [lib, actions]
}

rule "no-supabase-ssr-in-client-layers" {
  severity    error
  description "Supabase SSR server client must not be in components or hooks"
  forbid import "@supabase/ssr"
  except in   [lib, actions, app]
}

rule "no-db-client-in-components" {
  severity    error
  description "Components must not import Prisma db client — use actions"
  forbid import "@/lib/db"
  except in   [actions, lib]
}

rule "no-polar-in-components" {
  severity    error
  description "Polar payment logic must stay in lib/actions"
  forbid import "@/lib/polar"
  except in   [actions, lib, app]
}

rule "no-process-env-scatter" {
  severity    error
  description "All env vars through src/config only"
  forbid pattern "process.env."
  in layers   [actions, hooks, components, app]
}

rule "no-next-router" {
  severity    error
  description "Use next/navigation not next/router in App Router"
  forbid import "next/router"
  except in   []
}

rule "no-any" {
  severity    warning
  description "No TypeScript any — use proper types from src/types"
  forbid pattern ": any"
  in layers   [types, actions, lib, hooks]
}

rule "no-console" {
  severity    warning
  description "Remove console.log before shipping"
  forbid pattern "console.log"
  in layers   [components, hooks, app, actions]
}

# ── Enforcements ──────────────────────────────────────────────

enforce no-circular-imports
enforce no-dead-exports

# ── Features ──────────────────────────────────────────────────

features {

  feature "auth" {
    description  "Google OAuth via Supabase. DB trigger auto-creates profiles row on first sign-in."
    status       built
    files        ["src/lib/supabase.ts", "src/app/api/auth/callback/route.ts", "src/app/(auth)/login/page.tsx"]
    routes       ["/login", "/logout", "/api/auth/callback"]
    tables       ["auth.users", "public.profiles"]
    notes        "DB trigger creates profiles row — never create profiles rows manually in code"
  }

  feature "notes-crud" {
    description  "Full CRUD on notes. Tags, pinning, public/private, linked notes."
    status       built
    actions      ["getNotes", "getNoteById", "createNote", "updateNote", "deleteNote", "togglePin", "togglePublic"]
    hooks        ["useNotes", "useNote"]
    components   ["NoteList", "NoteEditor", "NoteCard", "NoteToolbar"]
    routes       ["/dashboard", "/note/[id]"]
    tables       ["public.notes"]
    depends-on   ["auth"]
  }

  feature "note-history" {
    description  "Version snapshots of note content. Manual and auto-triggered saves."
    status       built
    actions      ["createSnapshot", "getSnapshots", "restoreSnapshot"]
    hooks        ["useNoteHistory"]
    components   ["HistoryPanel", "SnapshotCard"]
    tables       ["public.note_versions", "public.notes_history_snapshots"]
    depends-on   ["notes-crud"]
    notes        "Snapshots have a mode (plain/rich) and trigger (manual/auto) field"
  }

  feature "payments" {
    description  "Polar.sh one-time purchase. Webhook sets profiles.is_paid = true."
    status       built
    files        ["src/lib/polar.ts", "src/app/api/webhooks/polar/route.ts"]
    actions      ["updatePaymentStatus", "getPaymentStatus"]
    routes       ["/upgrade", "/api/webhooks/polar"]
    tables       ["public.profiles"]
    notes        "is_paid boolean is the gate — check this field, not current_plan. polar_customer_id links to Polar."
  }

  feature "access-control" {
    description  "Protected routes check profiles.is_paid. Redirects to /upgrade if false."
    status       built
    files        ["src/app/(protected)/layout.tsx", "src/actions/profiles.ts"]
    depends-on   ["auth", "payments"]
    notes        "All gating logic in (protected)/layout.tsx — do not duplicate gating in individual pages"
  }

  feature "public-notes" {
    description  "Notes with is_public=true viewable at /note/[id] without authentication."
    status       built
    components   ["PublicNoteView", "PublicNoteHeader"]
    routes       ["/note/[id]"]

  feature "feedback" {
```

### 11.6 Convention example generation

For the conventions table, generate a concrete example by substituting tokens:

```javascript
function resolveExample(pattern, name) {
  // pattern: "src/actions/{domain}.ts", name: "notes"
  // → "src/actions/notes.ts"

  // pattern: "src/hooks/use{Name}.ts", name: "notes"
  // → "src/hooks/useNotes.ts"  (capitalize first char of name)

  // pattern: "src/components/app/{Name}.tsx", name: "noteCard"
  // → "src/components/app/NoteCard.tsx"

  return pattern
    .replace('{domain}', name.toLowerCase())
    .replace('{Name}', name.charAt(0).toUpperCase() + name.slice(1))
    .replace('{name}', name)
}
```

Use a fixed example name (`"notes"`) when generating the conventions table.

### 11.10 Backward compatibility implementation

In `parser.js`, the top-level dispatch must:

```javascript
// When encountering an unknown top-level keyword:
// Log to debug (not stderr) and skip until next top-level block
// A "top-level block" starts at column 0 with a keyword
// Skip logic: advance until RBRACE depth returns to 0, then continue
```

In all `parse*Block()` methods:

```javascript
// When encountering an unknown keyword inside a block:
// advance() once (skip the keyword)
// if next token is STRING or IDENTIFIER: advance() once more (skip the value)
// if next token is LBRACKET: skip entire list
// continue parsing — do not throw
```

This guarantees any `.arc` file written for arc v1 will parse cleanly in any future version.
